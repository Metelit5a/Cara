

# Project CARA: From Academic Prototype to Real-World Robustness

## Executive Summary

This report outlines the architectural evolution and data engineering breakthroughs achieved in Project CARA, a computer vision system designed for automated facial skin analysis. Initial iterations of the project relied on five distinct models, which, despite achieving near-perfect validation scores, demonstrated severe catastrophic failure when exposed to real-world user photos.

Through rigorous root-cause analysis, we identified that the models were learning dataset signatures and artifacts rather than genuine clinical features. By restructuring the machine learning architecture, enforcing strict training-inference pipeline parity, and employing creative domain adaptation via data repurposing, we successfully transformed CARA into a highly accurate, robust system capable of handling "in-the-wild" images.

## 1. System Architecture & The Data Reset

The initial approach of using five models trained on disparate, low-quality datasets (e.g., attempting classification on object-detection datasets) proved fundamentally flawed. We executed a complete architectural reset, standardizing on three **EfficientNetB0** models running behind a rule-based Business Logic Processor (BLP) engine:

* **Acne Severity:** Single-label (Clear / Mild / Moderate / Severe)
* **Skin Type:** Single-label (Oily / Dry / Normal / Combination)
* **Skin Conditions:** Multi-label (Pores / Blackheads)

These models now feed deterministic predictions into a JSON-configured BLP engine, which maps the findings to specific ingredient recommendations. To optimize our iteration speed, we also introduced a checkpoint mechanism, allowing us to retrain from saved weights rather than starting from scratch.

## 2. Establishing Real-World Guardrails

We discovered that standard validation testing (`test_model_accuracy.py`) merely proved the models could memorize their training distributions. To prevent future regressions, we engineered a three-layered real-world regression suite (`test_real_world_regression.py`):

1. **Per-Image Assertions:** Automated testing against hand-labeled ground truth using real user photos.
2. **Population-Level Sanity Checks:** Validations to catch class-collapse (e.g., ensuring the model predicts `healthy` at least once on a batch of predominantly clean faces).
3. **Multi-Label Specific Checks:** Added to support our architectural shift in condition tracking.

## 3. Enforcing Pipeline Parity & Bug Eradication

A critical breakthrough occurred while debugging a perfectly clean selfie that the system flagged as "severe acne, oily, with wrinkles." Root-cause analysis revealed a massive divergence between our training data and our production inference pipeline:

* **Dead Code in Preprocessing:** A face-crop pipeline existed but was entirely unlinked. The models were trained on tight face crops but were being fed full images (including backgrounds) in production. We wired the `PreprocessingPipeline` into the `AnalysisService`, making face detection mandatory (`require_face=True`).
* **EXIF & Rotation Handling:** We identified that `cv2.imdecode` ignored EXIF orientation tags, causing iPhone photos to load sideways and fail the Haar cascade detectors. We resolved this by decoding with PIL, applying `ImageOps.exif_transpose()`, and engineering a rotational retry mechanism (±15°, ±30°, ±45°, ±90°, 180°) to ensure faces are always cropped upright.
* **Codebase Sanitization:** We aggressively purged orphaned modules and dead code to ensure architectural clarity for future contributors.

## 4. Architectural Reframing: Single-Label to Multi-Label

Our Skin Conditions model continuously hallucinated issues on clean faces because it was constrained by a multi-class softmax output—it was mathematically forced to pick a condition and could not output "none."

We reframed the problem to match the biological reality of human skin, shifting to a multi-label classification architecture:

| Feature | Previous (Single-Label) | Current (Multi-Label) |
| --- | --- | --- |
| **Loss Function** | CrossEntropyLoss | BCEWithLogitsLoss |
| **Output Shape** | Softmax (Sum to 1) | Sigmoid (Independent probabilities) |
| **Prediction** | Argmax | Independent thresholds per class |
| **Label Format** | Single Integer | Multi-hot vector `[1, 0, 1]` |
| **Null Output?** | No (Forced to pick) | **Yes (Can output "None")** |

We discarded clinically unhelpful classes (e.g., "wrinkles" learned from elderly stock photos) and focused strictly on pores and blackheads. To provide the necessary "negative anchors" for the BCE loss, we repurposed **483 curated clear-skin faces** from the Acne model's dataset, achieving zero-cost data expansion.

## 5. Domain Adaptation via Creative Data Repurposing

Despite the multi-label shift, regression tests revealed a new bias: the model over-fired on internet stock photos. It had learned the photographic signature of our datasets, equating the high-quality ACNE04 dataset style with "clean" and arbitrary Roboflow internet photos with "blackheads."

Instead of undertaking a massive, time-consuming data labeling effort, we utilized domain adaptation via auxiliary data. We extracted **1,500 carefully sampled images** from our orthogonal `skin_type` dataset (biased towards "normal" and "dry" classes to minimize label noise).

By injecting these into our Skin Conditions negative folder, we successfully broke the model's reliance on photographic style, teaching it to look for actual pores rather than lighting conditions. Our held-out test split now achieves a macro-F1 score of **0.996**.

## Core Generalizable Lessons

1. **Data Quality > Model Complexity:** A perfectly trained model on flawed or mislabeled data will simply be perfectly wrong.
2. **Training vs. Real-World Accuracy:** Validation splits of training data only prove memorization. True performance can only be measured against a held-out distribution matched to production realities.
3. **Architectural Alignment:** The loss function and output shape must reflect the physical reality of the problem domain (e.g., using multi-label sigmoid when conditions can co-occur or be entirely absent).
4. **Preprocessing Parity is Mandatory:** Any divergence between how an image is processed during training and inference will result in catastrophic failure in production.
